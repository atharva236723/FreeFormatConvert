# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

A client-side file converter (image / video / audio / documents) built on Astro. There is no backend — every conversion runs in the visitor's browser. This is the product's core pitch, not an implementation detail: files never leave the device, unlike server-upload competitors.

`FEATURES.md` is the canonical catalog of *what the app can do* (user-facing feature list); this file covers *how it's built*. `DESIGN.md` (see Styling) is the source of truth for visual decisions. `README.md` is the short public-facing intro. Keep `FEATURES.md` in sync when you add or remove a conversion/tool.

## Commands

```sh
npm install       # install dependencies
npm run dev       # start dev server at localhost:4321
npm run build     # production build to ./dist/ (astro build — transpiles TS, does NOT full-type-check)
npm run preview   # preview the production build locally
npx astro check   # full TypeScript type-check (via @astrojs/check) — run this separately before shipping
npm run astro ... # run any Astro CLI command, e.g. `npm run astro add react`
```

`npm run build` is just `astro build`, which transpiles TS with esbuild and only type-checks `.astro` files — it will NOT catch type errors in `.ts` files. Run `npx astro check` for that (`@astrojs/check` + `typescript` are devDependencies). There is no linter or automated test suite configured. Requires Node >=22.12.0 (see `engines` in `package.json`).

When starting the dev server, use background mode so it doesn't block:
```sh
astro dev --background
```
Manage it with `astro dev stop`, `astro dev status`, and `astro dev logs`.

### Deployment

Deployed to **Cloudflare Workers** as a **static-assets** site — there's no SSR adapter because the whole app is static (every route is pre-rendered at build time, including the `sitemap.xml.ts` endpoint). `wrangler.jsonc` (Workers config, not Pages) points `assets.directory` at `./dist`, with `html_handling: "auto-trailing-slash"` (matches Astro's directory output, so `/png-to-jpg` serves `/png-to-jpg/index.html` and `/png-to-jpg` → 307 → `/png-to-jpg/`) and `not_found_handling: "404-page"` (unmatched paths serve the built `404.html` with a real 404). `wrangler` is a pinned devDependency; `.wrangler/` local state is gitignored.

```sh
npm run build && npx wrangler deploy   # rebuild dist/ then push all assets to the Worker
```

`dist/` is gitignored, so a fresh clone must `npm install && npm run build` before deploying. There is no `deploy` npm script — run `npx wrangler deploy` directly.

**Domain caveat (important):** `site` in `astro.config.mjs` is hard-coded to `https://freeformatconvert.com`, and every canonical `<link>`, the OG URLs, the generated `sitemap.xml`, and the `Sitemap:` line in `public/robots.txt` derive from it. That is **not** the `*.workers.dev` URL the Worker serves on until a custom domain is attached — so until `freeformatconvert.com` is pointed at this Worker, the sitemap/canonicals reference a host that doesn't resolve. Correct once the domain is live; misleading before then. If you deploy under a different hostname, update `site` **and** the absolute URL in `public/robots.txt` together (see the sitemap/robots note under Page composition).

## Architecture

### Conversion engine (the core of the app)

Three engines are picked automatically per conversion — the split is invisible to the user, it exists purely for speed and to keep heavy code lazy:

- **`src/lib/converter/imageEngine.ts`** — Canvas API fast path (`createImageBitmap` → draw → `canvas.toBlob`) for raster pairs the browser can actually encode. Instant, no network request. `decodeImage()` falls back to an `<img>` element when `createImageBitmap` throws — this is what makes **SVG** sources work (`createImageBitmap` can't decode SVG; an `<img>` rasterizes it, defaulting to 512² when the SVG has no intrinsic size). `decodeImage()` also **transcodes HEIC/HEIF to JPEG via `heic2any` (dynamically imported, so it stays out of the eager bundle) before decoding** — neither `createImageBitmap` nor the ffmpeg core can read HEIC, so this is the *only* path that can, and it's what makes `heic → jpg/png/webp` (and `heic → ico`) work at all. Because of that, `isFastPathPair()` treats HEIC as a Canvas-decodable source, and `getAvailableTargets` curates a HEIC source's targets down to the Canvas-encodable rasters + ICO + PDF (dropping GIF/BMP/TIFF, which would need ffmpeg to decode the HEIC and fail). Also exports **`convertImageToIco()`**: ffmpeg's wasm core has no ICO muxer, so ICO is encoded here by hand (rasterize → PNG downscaled to fit 256² → wrap in a single-entry ICO container with a PNG payload). `index.ts` routes any `image → ico` here regardless of source.
- **`src/lib/converter/ffmpegEngine.ts`** — `@ffmpeg/ffmpeg` wrapper for all audio, all video, and image formats Canvas can't handle. The ~30MB wasm core is **not bundled** — it's fetched lazily from jsDelivr (`https://cdn.jsdelivr.net/npm/@ffmpeg/core@0.12.10/dist/esm/`) the first time it's actually needed, then cached as a module-level singleton for the session. Uses the single-threaded core (not `core-mt`) so it works without COOP/COEP headers. If you bump `@ffmpeg/ffmpeg`, check that the pinned `CORE_VERSION` constant is still a compatible pairing — mismatches between the JS wrapper and the CDN-loaded core are a common ffmpeg.wasm failure mode. The `ffmpeg.exec()` timeout scales with input size (`computeTimeoutMs`: a base plus a per-MB allowance, capped at `MAX_TIMEOUT_MS`) instead of a flat constant — large-but-legitimate conversions near the size cap get more room, while a small stuck file still fails fast. Target-specific codec quirks are hard-coded in `transcode()` and each has a reason attached — don't strip them: (1) **webm** is pinned to VP8 (`-c:v libvpx`, the core's default VP9 encoder reliably traps with a WASM out-of-bounds fault) with `-pix_fmt yuv420p` (VP8 + alpha + `auto_alt_ref`, e.g. a transparent GIF, fails to open the encoder), `-deadline realtime -cpu-used 8 -b:v 1M` (libvpx's default speed blows the timeout on even a 10s clip in this single-threaded core), and `-c:a libvorbis` (the other default webm audio codec, Opus, traps the same way VP9 does); (2) the **`H264_TARGETS`** containers (`mp4`/`mov`/`m4v`/`mkv`/`3gp`) are forced to `-c:v libx264 -preset ultrafast -pix_fmt yuv420p` plus an even-dimension scale filter — `ultrafast` because libx264's default preset is ~40–60s/short-clip here and occasionally OOMs; the explicit `libx264` because `.3gp` otherwise defaults to H.263 (which rejects arbitrary sizes like 640×360); yuv420p + even dimensions because animated GIFs are frequently odd-sized and h264 rejects odd dimensions; (3) `apng` targets force the `apng` muxer with `-plays 0`, otherwise the `.apng` extension selects the single-frame PNG encoder and animation is lost. The tradeoff of `ultrafast`/`realtime` is larger output files in exchange for staying inside the timeout — the right call for a browser-only encoder. **This core (`@ffmpeg/core@0.12.10`) has no AV1, Opus-encode, AMR, or standalone-ALAC encoder** — the formats those would target are marked `inputOnly` in `formats.ts` rather than offered as dead conversions (see below). Besides `transcode()`, this module exposes two escape hatches reused by the standalone tools: **`runFFmpeg(file, buildArgs, outExt, opts)`** (arbitrary arg lists — see `mediaTools.ts`) and **`probeDuration(file)`** (runs ffmpeg with no output file and parses the `Duration:` banner from the log to get a media length without a full decode — the trim tool's fallback for containers the browser can't play).
- **`src/lib/converter/documentEngine.ts`** — handles the document + ebook conversions: **DOCX→PDF** (`mammoth` extracts HTML, then a manual block-layout pass writes a real text-selectable PDF via `jsPDF` — not html2canvas rasterization), **PDF→JPG/PNG** (`pdfjs-dist` renders each page to a canvas via the shared `renderPdfPage`/`pdfDataToImages` helpers; single page → one image, multiple pages → a `jszip` archive of numbered images), **DOCX→JPG/PNG** (no direct client-side Word renderer exists, so it routes through the same DOCX→PDF layout pass — `docxToPdfDoc` returns the jsPDF instance — then rasterizes the resulting PDF bytes through `pdfDataToImages`; single page → one image, multi-page → a zip), **image→PDF** (`jsPDF`, one image per page, flattened onto white so transparent PNGs don't go black — HEIC/HEIF sources are transcoded to JPEG with `heic2any` first, since no browser decodes HEIC via `createImageBitmap`), **PDF→Word (DOCX)** and **PDF→EPUB** (`pdfjs-dist` extracts text per page via `getTextContent()`/`hasEOL`, reflowed into paragraphs by `itemsToParagraphs`, then written to an editable `.docx` with the `docx` library or packaged into a spec-valid EPUB 3 zip via `jszip`; **PDF→Word additionally handles scanned/image-only PDFs** — a page with *no extractable text* is rendered to a PNG and embedded via `docx`'s `ImageRun` so the page still converts instead of producing a blank paragraph, which is what "scanned PDF won't convert to Word" used to mean), and **EPUB→PDF** (`jszip` unzips the ebook, walks the OPF `<spine>` in reading order, inlines each XHTML's images as data URLs, and flows the combined markup through the same `renderHtmlToPdf` block-layout pass). The libraries (`jspdf`, `pdfjs-dist`, `mammoth`, `jszip`, `docx`, `heic2any`) are all dynamically imported inside this module so they stay out of the eager bundle, exactly like ffmpeg's core. `renderHtmlToPdf` recurses into structural container tags (`div`/`section`/…) so deeply-nested EPUB bodies aren't flattened into one paragraph, while mammoth's already-flat output is unaffected. Deliberate quality bounds, not TODOs: **PDF→Word (text pages) and PDF↔EPUB are text-only** — they extract and reflow text, and do NOT reproduce the source's exact layout, columns, fonts, or image positions (there's no reliable fully-client-side way to do that; the scanned-page image fallback is the one exception, and it embeds the page as a picture rather than reconstructing layout); **PPTX→PDF stays excluded** for the same reason (no client-side renderer); and DOCX→PDF/EPUB→PDF use jsPDF's built-in Helvetica which **only covers Latin-1** (non-Latin scripts render blank until a Unicode font is embedded). `mammoth` is imported from its prebuilt browser bundle (`mammoth/mammoth.browser.js`) to avoid its Node-only deps; that subpath has no types, so a `declare module` shim lives in `src/env.d.ts`.

**pdfjs loading is centralized in `src/lib/converter/pdfjsLoader.ts`** — the single `loadPdfjs()` used by both `documentEngine.ts` and `tools/pdfTools.ts` (don't reintroduce a per-file `loadPdfjs`). It still `await import('pdfjs-dist')` to stay lazy, but it wires the worker as a **`GlobalWorkerOptions.workerPort`** (a cached module worker built via `new Worker(new URL('./pdfjsWorker.ts', import.meta.url), { type: 'module' })`), **not** the old `workerSrc` + `?url` string. The reason is a **Safari < 17.4 (older macOS/iOS) compatibility fix**: `pdfjs-dist@6` calls `Promise.withResolvers()` on both the main thread and inside its worker, and that method doesn't exist on those Safari versions — the failure surfaces mid-conversion as the cryptic `undefined is not a function (near '...e of t...')` and breaks *every* PDF path. So `pdfjsLoader.ts` imports the side-effect polyfill **`src/lib/converter/pdfWorkerPolyfill.ts`** (installs `Promise.withResolvers` when missing) for the main thread, and the worker entry **`src/lib/converter/pdfjsWorker.ts`** imports that same polyfill *before* `pdfjs-dist/build/pdf.worker.min.mjs` (ES module evaluation order guarantees the polyfill lands first — the worker is a separate JS realm the main-thread polyfill can't reach). Even pdfjs's `legacy` build uses `withResolvers`, so downgrading is not an alternative. The cached worker is safe to reuse because pdfjs never terminates an externally-supplied `workerPort` on `loadingTask.destroy()`. Don't revert to `workerSrc`/`?url` — it drops the worker-side polyfill and the Safari bug returns.

**`src/lib/converter/index.ts`** is the orchestrator every caller should use (`convertFile(file, targetExt, opts)`). It checks `getDocumentOp()` first and routes document conversions to `documentEngine`; otherwise it routes any `image → ico` to `imageEngine`'s Canvas ICO encoder (ffmpeg can't mux ICO), then decides Canvas fast-path vs ffmpeg via `isFastPathPair()`, infers video→audio extraction (adds `-vn` to skip video decoding entirely) by comparing source/target category, rejects files over the per-category `MAX_FILE_SIZE_BYTES` cap before doing any work, and normalizes every failure into a typed `ConversionError` (`reason: 'unsupported-pair' | 'timeout' | 'engine-load-failed' | 'file-too-large' | 'unknown'`). Note document conversions can change the output extension/count (multi-page PDF→image yields a `.zip`), so the document path returns its own `filename` rather than letting `convertFile` derive it from `targetExt`.

`ConversionError` lives in its own file, **`src/lib/converter/errors.ts`**, deliberately separate from the engines. `index.ts` needs the error class at the top level but must NOT statically import anything from `ffmpegEngine.ts` **or `documentEngine.ts`**, or Vite will pull their dynamic `import(...)` calls into the eagerly-loaded chunk graph and defeat the lazy-loading (Vite warns `INEFFECTIVE_DYNAMIC_IMPORT` if this regresses; the build should stay warning-free). Both engines are only ever reached via `await import('./…')` inside `convertFile`.

**`src/lib/formats.ts`** is the single source of truth for supported formats: per-category (`image`/`audio`/`video`/`document`) `FormatDef` lists, `detectCategory(file)`, `getAvailableTargets(category, ext)`, `getDocumentOp(sourceExt, targetExt)`, `isFastPathPair()`, `fileMatchesSourceExt(filename, expectedSourceExt)` / `normalizeExt(ext)` (source-file validation for the exact-pair pages — alias-aware via the small `EXT_ALIASES` map, so `.jpeg`/`.jfif` count as `jpg` and `.tif` as `tiff`), and `MAX_FILE_SIZE_BYTES` (per-category upload caps — image 50MB, audio 300MB, video 1GB, document 100MB — that bound worst-case conversion time rather than acting as a paywall). When adding a format, add it here — nothing else hardcodes a format list. Cross-category quirks live here too: `getAvailableTargets` merges the video→audio "extract audio" bonus targets, appends **PDF** as a target for image sources in `IMAGE_TO_PDF_SOURCES` (the Canvas-decodable set **plus `heic`**, which documentEngine decodes via heic2any), and for `document` sources returns a *curated* set (PDF→[JPG, PNG, Word/DOCX, EPUB], DOCX→[PDF, JPG, PNG], EPUB→[PDF]) rather than the media model's any-pair-within-a-category. `epub` is a `document` format (both source and target); `docx` stays `inputOnly` and is surfaced as a target only inside `getDocumentTargets` for PDF sources. DOCX→JPG/PNG are the `docx-to-image` `DocumentOp` (a DOCX→PDF→raster pipeline in documentEngine). Several media formats are also `inputOnly` — readable but never emitted, because writing them is either pointless or unreliable. Two reasons for the flag: **(a) redundant/rarely-wanted targets** — `jfif` (byte-identical to JPEG, so it's in `CANVAS_DECODABLE_EXTS` and converts via the Canvas fast path, but we never write a `.jfif` when `.jpg` is the canonical name for the same bytes) and the MPEG-TS / camcorder / DVD video containers `mts` / `m2ts` / `ts` / `vob` / `3g2` / `f4v` (ffmpeg demuxes them reliably so `mts-to-mp4` etc. are real conversions, but muxing *back out* to them is rarely wanted and less reliable in the wasm core); **(b) no working encoder in this ffmpeg core** — `avif` (browser decodes it via `createImageBitmap`, but there's no AV1 encoder, so `avif-to-jpg` works while `*-to-avif` can't), `opus` (encoder traps with a WASM out-of-bounds), `amr` (no encoder compiled in), and `alac` (no standalone `.alac` muxer — it only lives inside `.m4a`, which is already offered). The (b) formats stay valid *sources* (`opus-to-mp3`, `avif-to-png`, …) — `inputOnly` only removes them as targets. Don't un-flag them without confirming the CDN core actually gained the encoder. `converter-controller.ts` also checks `MAX_FILE_SIZE_BYTES` at file-selection time, before rendering the format picker, so oversized files get instant feedback instead of waiting to hit the check inside `convertFile`.

Known constraint baked into `isFastPathPair`: `canvas.toBlob()` cannot actually encode BMP or GIF in any browser (it silently falls back to PNG). The fast path is intentionally narrow — decodable-source set is broad, encodable-target set is only `jpg/jpeg/png/webp`. Anything else routes through ffmpeg, which has real encoders for those containers.

That same silent fallback is why the `canvasToBlob` helpers in **`imageEngine.ts` and `tools/imageTools.ts`** reject when `blob.type !==` the requested mime (not just when the blob is null). Per spec, a browser that can't encode the requested type produces an `image/png` blob and reports the real type — so on **Safari < 14** (older macOS/iOS, no WebP *encode* support) a `*-to-webp` conversion would otherwise hand back a PNG saved under a `.webp` name (a corrupt, mislabeled file with no error). The type-mismatch check turns that into an honest "this browser can't encode WEBP" error. Don't drop it back to a null-only check — `png/jpeg` are universally encodable so it never misfires there, and it's the only thing standing between old-Safari WebP users and a broken download.

**Animated images (GIF, APNG) can convert to video** and this is a load-bearing subtlety. Their MIME (`image/gif` / `image/apng`) makes `detectCategory` classify them as `image`, so a naive image target list would never offer MP4/WebM. `getAvailableTargets` therefore widens the target set for the exts in `ANIMATED_IMAGE_EXTS` to include the video containers, which is what makes `gif-to-mp4` (and the whole `gif → video` family) real conversions rather than dead pages. The invariant to preserve: `getAvailableTargets('image', 'gif')` must return the *same* targets at build time (when the SEO pages below pre-compute a preselected target) and at run time (when `converter-controller.ts` re-renders the picker after detecting the uploaded file's category). If those ever diverge, a page advertises a conversion its own picker won't offer — that exact bug is why this branch exists.

`formats.ts` also owns the data behind the per-conversion SEO pages (see Page composition): `getAllConversionPairs()` expands every `(source ext → target ext)` into a unique-slug list (`gif` lives in both the image and video lists — the pairs are deduped by slug, image wins, and the run-time category re-detection means one `gif-to-webp` page serves both), plus `getRelatedPairs()`, `getPopularPairs()` (a hand-curated slug list, filtered to slugs that actually generate), and `displayLabel()` (strips parentheticals so "GIF (animated)" → "GIF" for headings).

### UI

No UI framework is installed; every interactive piece is hand-written vanilla TS (see also `theme-toggle.ts` under Theming and the `nav-dropdown.ts` "Conversions" menu under Page composition). The converter widget is a custom element:

- **`src/components/Converter.astro`** — static markup/CSS. Visibility of each stage (`dropzone`, pick/converting/done/error) is driven entirely by a `data-state` attribute on the `<file-converter>` root, toggled via CSS descendant selectors (`file-converter[data-state="..."] .stage-x { display: block }`) — no framework, no layout-shift. Accepts optional `presetTarget` / `presetSource` / `presetSourceLabel` / `presetTargetLabel` props (used by the per-conversion SEO pages), each emitted as a `data-preset-*` attribute; `presetTarget` preselects the output and the dropzone copy is tailored server-side ("Drop your PNG file…"). **Styling gotcha:** the format-picker chips (`.format-chip`) are created at runtime by `converter-controller.ts`, so they never receive Astro's scoped `data-astro-cid` attribute — their CSS in this component's scoped `<style>` must be wrapped in `:global(...)` or it silently won't apply (the chips then render as unstyled default buttons). Same rule for any other element the controller injects.
- **`src/scripts/converter-controller.ts`** — `customElements.define('file-converter', ...)`, the whole state machine (`idle → dragging → file-selected → converting → done/error`), wired to drag/drop, the hidden file input, and dynamically-rendered format-picker buttons (built at runtime from `formats.ts`, not duplicated as static HTML per category). When `data-preset-target` is present, the matching format chip is auto-selected after a file is chosen (expanding the collapsed "Show all formats" group if the target lives inside it), so on a `/png-to-jpg`-style page the user only has to pick a file and hit Convert. **Source validation:** when `data-preset-source` is present (only the exact-pair pages set it), `selectFile()` rejects any dropped file whose format isn't that source — via `fileMatchesSourceExt()` in `formats.ts` — and shows a mismatch error plus an escape link to the homepage all-in-one converter. The homepage (`<Converter />` with no presets) and the category hubs deliberately omit `presetSource`, so they stay fully auto-detecting: any file is accepted and the picker is built from its detected category.

Follow this pattern for any other non-trivial interactive component in this project rather than reaching for React/Vue — there's no framework integration configured, and adding one for a single self-contained widget isn't worth the bundle cost next to the already-lazy-loaded wasm core. The NavBar "Conversions" and "Tools" dropdowns (`nav-dropdown.ts`) are a second, simpler instance of the same approach: a plain script that initializes any `[data-dropdown]` root (click to toggle, outside-click / Escape to close, `aria-expanded` synced). It keeps a **shared registry of every dropdown's close handler** so opening one closes the others — the triggers `stopPropagation` on click, so they'd otherwise never reach a sibling's outside-click listener and two menus could sit open and overlapping. Note the panels are anchored to the **`.nav-inner` container's right edge** (`.nav-dropdown` is `position: static`, panel is `right: 0`), not to the trigger: a ~600px mega-menu opened from a right-of-center trigger would run off the viewport otherwise. The `max-width: 599px` breakpoint switches the panel to centered (`left: 50%` / `translateX(-50%)`).

### Standalone tools (compress / edit / PDF / convert)

Alongside the format converter there's a second family of single-purpose **tools** (freeconvert.com-style: compressors, image editors, PDF utilities, unit converters). They share the converter's "nothing leaves your device" story — every one runs fully client-side, no API/LLM. They live apart from the `convertFile` pipeline because they're interactive editors (sliders, crop boxes, page grids) rather than one-shot conversions.

- **`src/lib/tools/imageTools.ts`** — pure Canvas ops (`resizeImage`, `cropImage`, `rotateImage`, `flipImage`, `compressImage`, `readDimensions`). Reuses the same `createImageBitmap`→`<img>` decode fallback as `imageEngine.ts` (so SVG works). Encoding is limited to what `canvas.toBlob` can produce — `pickOutput` preserves JPEG/PNG/WebP and falls back to PNG for everything else; JPEG output flattens onto white so transparency isn't blackened. **Canvas only ever sees the first frame of an animated GIF**, so these functions would silently flatten one to a still image — animated GIFs are handled separately (see `mediaTools`'s `*Gif` helpers and the controller routing below), not here. Also exports **`compressImageToTarget(file, { targetBytes, mime })`** — the "compress to a target file size" path: it decodes once, then for each scale step (full size first) binary-searches the encode quality for the highest quality that fits under `targetBytes`, only downscaling when even the lowest quality overshoots. It's the one target-size tool that lands *exactly* under the target (Canvas re-encode is cheap, so it can iterate); it returns `underTarget: false` when the target is smaller than the tiniest achievable encoding.
- **`src/lib/tools/mediaTools.ts`** — ffmpeg-backed `compressVideo` / `compressAudio` / `compressGif` / `trimVideo`, plus the animation-preserving image edits **`resizeGif` / `cropGif` / `flipGif` / `rotateGif`** (all built on a shared `transformGif` that pipes the transform through a `split`/`palettegen`/`paletteuse` graph so every frame survives; 90° rotations use lossless `transpose`, other angles grow the canvas with `fillcolor=none`). Reuses the ffmpeg singleton via **`ffmpegEngine.runFFmpeg(file, buildArgs, outExt, opts)`** (a low-level escape hatch beside `transcode`), so the wasm core stays lazy and codec quirks (libx264 ultrafast, etc.) stay consistent. Only ever dynamically imported from a controller. The **target-file-size variants** live here too and are deliberately *approximate* (unlike the exact image path): **`compressAudioToTarget`** and **`compressVideoToTarget`** derive a bitrate from the clip length (`bits = targetBytes×8 ÷ seconds`; video reserves a fixed audio budget and caps with `-maxrate`/`-bufsize`) — single-pass in the single-threaded wasm core, since accurate 2-pass is too slow, so they land *near* the target; **`compressGifToTarget`** has no bitrate knob at all, so it walks a scale/fps ladder from largest to smallest and stops at the first rung that fits (each rung is a full ffmpeg pass). All three resolve duration via `resolveDuration` — a media length the controller already read from an HTML `<video>`/`<audio>` element, falling back to `ffmpegEngine.probeDuration` for containers the browser can't play.
- **`src/lib/tools/pdfTools.ts`** — `pdf-lib`-backed `mergePdfs` / `extractPages` / `removePages` / `rotatePdf` / `splitToZip` (zip via `jszip`), plus `renderThumbnails` (pdfjs) for the selectable page grid. `pdf-lib` is a dependency added for these. Page numbers in this API are **1-based**. Dynamically imported only.
- **`src/lib/tools/units.ts`** — pure data + arithmetic conversion tables (length/mass/temperature/area/volume/speed/digital/time). Temperature uses to/from-base functions; everything else is a linear `factor`.

Each engine family is driven by **one mode-driven custom element + one controller**, not one-per-tool: `ImageTool.astro`/`image-tool-controller.ts` (`mode="resize|crop|rotate|flip|compress|colorpicker"`), `MediaTool.astro`/`media-tool-controller.ts` (`mode="compress-video|compress-audio|compress-gif|trim"`), `PdfTool.astro`/`pdf-tool-controller.ts` (`mode="merge|split|rotate|remove|extract"`), and `UnitConverter.astro`/`unit-converter-controller.ts` (optional `preset` category). They follow the same vanilla-TS custom-element pattern and `data-state` machine as `Converter.astro`, and share chrome via **`src/styles/tools.css`** (imported in each tool component's frontmatter; runtime-injected elements can't be Astro-scoped, same gotcha as the converter's `.format-chip`). Pages wrap the widget in **`ToolLanding.astro`** (hero + related links + FeatureStrip/HowItWorks, a tool-flavored sibling of `ConversionLanding.astro`); like the pair pages, it emits a JSON-LD `@graph` (**`SoftwareApplication`** + **`BreadcrumbList`** `Home › Tools › <heading>`) into the `head` slot for every tool, so a new tool page registered in `registry.ts` gets its structured data with no extra work. All four compress modes (`compress-image`/`-video`/`-audio`/`-gif`) render a **"By quality / Target size" mode toggle** built at runtime by the controller; picking "Target size" hides the quality/bitrate/scale controls and routes to the `*ToTarget` engine functions above. The target-size control is a **slider ("dragger")** that runs from a small floor (~5% of the input) up to the original file size, with a live value readout beside the label and an estimated-result line beneath it. For the media tools, `media-tool-controller.ts` shares a `buildTargetControls(origBytes, onModeChange)` helper whose estimate just echoes the chosen size (the single-pass encoder lands close, so a real preview would cost a full ffmpeg pass). The **image** compressor (`image-tool-controller.ts`) builds its own richer slider: as you drag it runs the real `compressImageToTarget` **debounced** and shows the *actual* predicted output size, staying honest via the `underTarget` flag ("Can't go this small — closest is ~X"). **Layout gotcha:** these sliders must live stacked inside a `.tool-field` (label + value in a `.tool-slider-head` header row, slider on its own full-width line) — putting the slider in a `.tool-row` collapses its width and floats the value label, which looks broken. The media controllers capture the clip duration from their preview `<video>`/`<audio>` on `loadedmetadata` and pass it into those functions so the bitrate math doesn't need a separate ffmpeg probe.

Two controller behaviours are load-bearing and easy to regress. **(1) Animated-GIF routing in `image-tool-controller.ts`:** on file select it counts Graphic Control Extension blocks (`21 F9 04`) to detect a multi-frame GIF; for resize/crop/rotate/flip it then dispatches to `mediaTools`'s `*Gif` helpers (ffmpeg) instead of the Canvas path and shows an "animation will be preserved" notice — otherwise the Canvas path silently emits a still `.png`. This is the one place an "image" tool reaches for the ~30MB ffmpeg core, and it's deliberate. Static images and non-GIFs keep the instant Canvas path. **(2) Trim duration fallback in `media-tool-controller.ts`:** the trim UI reads duration/seek from an HTML `<video>`, but the file-type check accepts containers Chrome can't decode (AVI/WMV/FLV, some MKV/MOV) where `<video>` fires `error` or never reports a duration. On that error — or a 5s stall — it calls **`ffmpegEngine.probeDuration(file)`** to read the length, swaps the dead preview for a note, and lets the trim proceed; browser-playable formats keep the live-preview fast path. Don't tighten the file-type checks to "fix" these — the fallbacks are the fix.

The tool routes (`/compress-image`, `/resize-image`, `/merge-pdf`, `/trim-video`, `/unit-converter`, …) are **standalone page files**, not matrix-generated. **`src/lib/tools/registry.ts`** is their single source of truth (`TOOL_GROUPS` + `TOOL_PATHS`); the `/tools` index, the NavBar "Tools" mega-menu, the Footer "Tools" column, and `sitemap.xml.ts` all read from it — add a tool page and register it there so it appears everywhere and isn't a dead sitemap entry. Heavy deps (`@ffmpeg/*`, `pdf-lib`, `pdfjs-dist`, `jszip`) stay lazy: the controllers `await import(...)` the tool libs, and the tool libs `await import(...)` the vendored packages, so nothing heavy lands in the eager bundle.

### Styling

**No Tailwind**, despite the `tailwind-4-docs` skill being available in `.agents/skills/` — it isn't used in this project. Styling is hand-written CSS using custom properties translated 1:1 from `DESIGN.md`'s token tables:
- `src/styles/tokens.css` — every color/spacing/radius/typography/elevation token as a `--variable`.
- `src/styles/global.css` — reset + base typography, imports `tokens.css`.

[DESIGN.md](DESIGN.md) (Vercel-inspired design system) is the source of truth for all visual decisions — colors, type scale, spacing, radius, elevation, component chrome. **Read it before writing or modifying any UI.**

### Fonts

Configured via Astro's native Fonts API in `astro.config.mjs` (`fonts: [...]` using `fontProviders.fontsource()`), rendered with `<Font cssVariable="..." />` from `astro:assets` in `src/layouts/Layout.astro`. Three faces: **Geist Sans** (body), **Geist Mono** (eyebrows/labels), and **Space Grotesk** (hero display headings). Astro downloads and self-hosts these at build time — there's no runtime request to a third party, which reinforces the "nothing leaves your device" story. Don't add a manual `<link>` tag or a Fontsource npm dependency; extend the `fonts` array in config instead. **Adding a font requires a dev-server restart** — the Fonts API fetches at startup, so HMR won't pick up a new entry.

Note the two-name indirection each font uses (mirrors Geist): the Fonts API defines a hashed variable named by `cssVariable` (`--font-geist-sans`, `--font-space-grotesk`), and `tokens.css` maps that to a stable consumer token (`--font-sans`, `--font-display`) with a fallback stack. Components reference the `tokens.css` name, never the raw `<Font>` variable. The hero `<h1>`s (in `index.astro` and `ConversionLanding.astro`) use `--font-display` with an animated gradient clipped to the text (`background-clip: text` + a `background-position` pan) — the loop is only seamless because `background-size` and the keyframe end position both equal `200%` and the gradient's first/last color stops match; changing one without the others reintroduces a visible jump at the loop point.

### Theming

Light/dark is driven by a `data-theme="dark"` attribute on `<html>`, with `tokens.css` redefining every `--color-*` / `--shadow-*` variable under a `[data-theme='dark']` block — components never branch on theme, they just consume tokens. Three pieces coordinate:
- An **inline `is:inline` script in `src/layouts/Layout.astro`** runs before first paint, reading `localStorage.theme` and **defaulting to dark for new visitors** (a returning user's explicit stored choice wins; the OS `prefers-color-scheme` is intentionally *not* consulted). This must stay inline and in `<head>` — moving it to a module or the body reintroduces a flash-of-wrong-theme.
- **`src/scripts/theme-toggle.ts`** wires the NavBar toggle button: flips the attribute, persists to `localStorage.theme`, and syncs `aria-pressed`. It's the only place the theme is written after load.
- The toggle lives in `NavBar.astro`; `localStorage.theme` (values `'light'`/`'dark'`) is the single persisted key.

Also in `Layout.astro`'s `<head>`: the **Google Analytics (gtag.js) tag** — measurement ID `G-GHKMYV9LM8`. Both the loader and the `gtag('config', ...)` snippet carry Astro's `is:inline` directive so they're emitted verbatim rather than bundled/processed (drop `is:inline` and Astro breaks the inline analytics snippet). It lives in the shared layout, so it loads on every page; there's no consent gate.

### Motion

A **site-wide motion layer** rides on top of every component's own transitions. It's a **progressive enhancement gated by an `is:inline` `<head>` script in `Layout.astro`** that adds `js-motion` + `page-enter` to `<html>` before first paint (so scroll-reveal is flash-free) and arms a **2.5s failsafe** that force-adds `motion-failsafe` if the reveal script never signals `window.__ffcMotionReady`. Four pieces coordinate — touch them together:
- **`src/scripts/motion.ts`** (loaded via `<script src>` at the end of `<body>`) — vanilla TS, no framework. `IntersectionObserver` reveals each `main > section` (excluding `.hero`, which has its own entrance) as it enters view; sections containing a known card grid (`STAGGER_GRIDS`) are opted out of the block reveal and instead tag their heading + each card individually with a capped `--reveal-i` stagger index for a cascade. Also eases same-page `#hash` anchor clicks into view. It sets `__ffcMotionReady = true` up front, and **early-returns without hiding anything when `prefers-reduced-motion` is set**.
- **`src/styles/global.css`** — the reveal CSS: `.js-motion:not(.motion-failsafe) main > section:not(.hero)` (and any `[data-reveal]`) start lowered + transparent and settle on `.is-revealed`; a `page-enter` body fade; a tactile `:active` press on buttons/links; and a `prefers-reduced-motion` block that force-reveals everything. The `js-motion` gate means **JS-disabled / failsafe states never leave content stuck hidden**.
- **`src/styles/tokens.css`** — shared easing/duration tokens (`--ease-smooth`/`--ease-out-soft`/`--ease-spring`, `--dur-fast`/`-base`/`-slow`/`-reveal`). Consume these instead of hand-writing `0.15s ease` per component.
- **Component-local entrances** reuse those tokens: `Converter.astro` eases each `data-state` stage in (`ffc-stage-in`), and `NavBar.astro` animates each dropdown panel open on `[hidden]` flip (`ffc-dropdown-in`, with a `translateX(-50%)`-preserving `-center` variant below the 599px breakpoint so the centered panel doesn't jump off-screen).

### Page composition

This is a multi-page app (Astro's default — every route is a full document, no client-side router). `src/pages/` holds one `.astro` file per route:
- `index.astro` — deliberately homepage-as-tool: `NavBar → Converter (the hero) → Popular conversions grid → FeatureStrip → HowItWorks → seo-content (long-form prose) → Faq → Footer`. The "Popular conversions" grid links into the per-conversion pages below (`getPopularPairs`). There is no separate marketing *splash* page ahead of the tool — don't reintroduce that split. (The per-conversion pages below are not that split: each one *is* a working converter, just with a specific heading and a preselected target.)
- **`[conversion].astro`** — a dynamic catch-all route whose `getStaticPaths()` statically generates **one page per conversion pair** from `getAllConversionPairs()` (~690 pages: `/png-to-jpg`, `/mp4-to-mp3`, `/gif-to-mp4`, …). This is the SEO surface — each page carries its own `<h1>`, meta title/description, and canonical, and mounts a `Converter` preset to that pair's target. It just wires props into `ConversionLanding.astro`; add a format to `formats.ts` and its pages appear automatically.
- **`conversions.astro`** — the `/conversions` index: every generated pair grouped by source category (Image / Video / Audio / Document, via `getAllConversionPairs()`) with a jump nav. Standalone file, so it's listed in `sitemap.xml.ts`'s `STATIC_PATHS`. Linked from the NavBar dropdown's "View all conversions" link.
- **`video-to-gif.astro` / `image-to-gif.astro`** — hand-written category **hub** pages. They aren't `(source ext → target ext)` pairs (there's no `video`/`image` ext), so they can't come from the matrix and are standalone files that also render `ConversionLanding.astro` with `presetTarget="gif"` and a curated related-links list. Add hubs as explicit files like these, and remember to also list them in `sitemap.xml.ts`'s `STATIC_PATHS` (the pair pages are added there automatically, standalone files are not).
- **`pdf-converter.astro` / `document-converter.astro` / `ebook-converter.astro`** — broader category hubs for the document/ebook conversions. Unlike the gif hubs these render `ConversionLanding.astro` with **no `presetTarget`** (its preset props are optional), so the visitor gets the general converter and picks any supported target after upload — they exist for the "PDF converter" / "document converter" / "ebook converter" search intent and cross-link the specific pairs. Also registered in `sitemap.xml.ts`. Note the PDF→Word pair page is at the ext-based slug **`/pdf-to-docx`** (there's no `word` ext); its `<h1>`/title still read "PDF to Word" because `displayLabel('Word (DOCX)')` → "Word", so link to `/pdf-to-docx` with a "PDF to Word" label rather than inventing a `/pdf-to-word` route.
- `about.astro`, `contact.astro`, `terms.astro`, `privacy.astro` — content/legal pages. **`contact.astro`** is the one page that talks to an external service: its form POSTs via `fetch` (AJAX, `Accept: application/json`) to a **Formspree** endpoint (`https://formspree.io/f/mgojpyvj`), swapping the form for an animated success card (a stroke-drawn checkmark) on `ok: true` and showing an inline error banner otherwise; a hidden `_gotcha` honeypot filters spam. The destination inbox is configured on Formspree's side and is deliberately **not** rendered anywhere on the page. This is the only user-initiated network POST in the app — it sends the typed message, never a file, so it doesn't break the files-never-leave-your-device story. If submissions aren't arriving, it's almost always a Formspree-side config issue (form not activated / recipient unverified / notifications off), not the code — the endpoint returning `{ ok: true }` means the submission was accepted.
- `404.astro`, `500.astro` — error pages (Astro emits `404.html`/`500.html` on build for the host to serve).
- **`sitemap.xml.ts`** — an endpoint that lists `STATIC_PATHS` + every pair slug. It (and the `<link rel="canonical">` / OG tags in `Layout.astro`) depend on `site` being set in `astro.config.mjs` (`https://freeformatconvert.com`) to build absolute URLs. The sitemap is *generated*, not a static file — register new standalone routes in `STATIC_PATHS` (pairs come from `formats.ts` automatically); don't add a hand-maintained `public/sitemap.xml`, which would go stale. `public/robots.txt` is a static file that allows all crawlers and points them at the absolute `https://freeformatconvert.com/sitemap.xml`; keep that URL in sync if the domain changes.

**`src/components/ConversionLanding.astro`** is the shared template behind both the generated pair pages and the hubs (heading + subtitle + `Converter` + related-links grid + `FeatureStrip`/`HowItWorks`). Its `presetTarget`/`presetSource`/`presetSourceLabel`/`presetTargetLabel` props are optional and forwarded straight to `Converter`: pair pages pass `presetSource`+`presetTarget` (from `[conversion].astro`, using the pair's `sourceExt`/`targetExt`) to both preselect the target and enable source validation, while category hubs omit them to show the general auto-detecting converter. Extend it rather than duplicating its markup into individual pages. **Per-pair structured data:** when both source+target labels are present (pair pages only, not the hubs) it emits a single JSON-LD `@graph` into the `head` slot combining a **`SoftwareApplication`** node (marks the page as a standalone free browser tool — rich-result eligibility), a **`BreadcrumbList`** (`Home › Conversions › <Src> to <Tgt>`, which is what renders the breadcrumb trail under the title in Google results), and the **`FAQPage`** (built from the same `faqs` array as the visible accordion — keep them in lockstep, same rule as `Faq.astro`). The URLs are absolutized via `new URL(canonical, Astro.site)`, so this too depends on `site` being set correctly in `astro.config.mjs` (see the domain caveat under Deployment). Adding a format to `formats.ts` gives its new pages this schema automatically — don't hand-add JSON-LD per pair page.

**`Faq.astro`** is mounted on `index.astro` (after the `seo-content` section) and is the **single source of truth for the homepage FAQ**: one `faqs` array drives both the visible `<details>` accordion and the `FAQPage` JSON-LD it emits (via `<script is:inline type="application/ld+json">`). Keep those in lockstep — Google's rich-results policy requires the structured data to match on-page content, so edit the `faqs` array rather than hand-writing schema. Because the component owns the FAQPage node, `index.astro`'s own `structuredData` `@graph` carries only the `WebApplication` entity (don't re-add a second FAQPage there, or the page ships two). Answers are HTML strings, rendered with `set:html` and styled via `:global()` selectors inside the component's scoped `<style>`.

Every page wraps its content in `Layout.astro` and shares `NavBar` + `Footer`. Both the `NavBar` and `Footer` link to the four content/legal pages (`/about`, `/contact`, `/terms`, `/privacy`) — keep those two link sets in sync. The NavBar's first item is a **"Conversions" mega-menu dropdown** (markup in `NavBar.astro`, behavior in `nav-dropdown.ts`): its columns are built server-side from `getPopularPairs()` grouped by source category, followed by the category-hub links and a "View all conversions" link to `/conversions` — so it stays in sync with `formats.ts` automatically. `Footer.astro` mirrors that same construction: a brand/tagline block plus link columns built server-side from `getPopularPairs()` grouped by source category (Images/Video/Audio/Documents), a "Tools" column of the category hubs + `/conversions`, and a "Company" column of the four content/legal links — so it also tracks `formats.ts` automatically and its legal links must stay in sync with the NavBar's. (Editing `Footer.astro`'s `<style>` can leave Vite serving a stale scoped-style module even after HMR reports the change and the SSR HTML is correct — restart the dev server if the footer looks unstyled.) There's still no JS hamburger for the plain links: below 600px they wrap onto their own centered second row (see the `max-width: 599px` block in `NavBar.astro`), where the dropdown panel also re-centers itself under that row instead of anchoring to the trigger. Content pages use `PageHeader.astro` for the title/subtitle band and the `.prose` / `.button-primary` utility classes defined in `global.css` — reach for those before writing new per-page CSS, so the pages stay visually consistent.

`Layout.astro` also mounts **`PageLoader.astro`** as the first child of `<body>` — a slim brand-gradient top progress bar that rides on top of the standard full-document (MPA) navigation. Its early `is:inline` script trickles the bar toward ~90% when an internal link is clicked (stashing a `ffc:navigating` flag in `sessionStorage`), then the *next* document reads that flag on first paint and snaps the bar to 100% and fades it out — a start-on-leave / finish-on-arrive effect with no client-side router. It ignores external links, `target=_blank`, `download`, hash-only and modifier-clicks, and resets cleanly on bfcache `pageshow`. Like the theme script and analytics, it's `is:inline` on purpose (must run before bundled scripts); don't convert it to a router.
