# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

A client-side file converter (image / video / audio / documents) built on Astro. There is no backend — every conversion runs in the visitor's browser. This is the product's core pitch, not an implementation detail: files never leave the device, unlike server-upload competitors.

### Branding (mid-migration — read before touching copy)

The user-facing chrome — `NavBar` logo, `Footer`, the favicon/PWA set in `public/` (`site.webmanifest` `name`, plus the `apple-mobile-web-app-title` meta in `Layout.astro`), and the `<title>` / `<meta name="description">` defaults in `Layout.astro` — all say "Free Format Convert". The main straggler still on the old **"Formart"** name is the `hello@formart.app` address / titles on `contact.astro`. Treat "Free Format Convert" as the correct name; when you touch a file that still says "Formart", migrate it, but don't do a blind repo-wide rename without confirming (the email domain may not be registered yet).

The favicon/manifest assets in `public/` are self-hosted (no third-party icon CDN), consistent with the "nothing leaves your device" story — replace the files in place rather than hotlinking.

## Commands

```sh
npm install       # install dependencies
npm run dev       # start dev server at localhost:4321
npm run build     # production build to ./dist/ (astro build — transpiles TS, does NOT full-type-check)
npm run preview   # preview the production build locally
npx astro check   # full TypeScript type-check (via @astrojs/check) — run this separately before shipping
npm run astro ... # run any Astro CLI command, e.g. `npm run astro add react`
```

`npm run build` is just `astro build`, which transpiles TS with esbuild and only type-checks `.astro` files — it will NOT catch type errors in `.ts` files. Run `npx astro check` for that (`@astrojs/check` + `typescript` are devDependencies). There is no linter or automated test suite configured. Requires Node >=22.12.0 (see `engines` in `package.json`).

When starting the dev server, use background mode so it doesn't block:
```sh
astro dev --background
```
Manage it with `astro dev stop`, `astro dev status`, and `astro dev logs`.

## Architecture

### Conversion engine (the core of the app)

Three engines are picked automatically per conversion — the split is invisible to the user, it exists purely for speed and to keep heavy code lazy:

- **`src/lib/converter/imageEngine.ts`** — Canvas API fast path (`createImageBitmap` → draw → `canvas.toBlob`) for raster pairs the browser can actually encode. Instant, no network request.
- **`src/lib/converter/ffmpegEngine.ts`** — `@ffmpeg/ffmpeg` wrapper for all audio, all video, and image formats Canvas can't handle. The ~30MB wasm core is **not bundled** — it's fetched lazily from jsDelivr (`https://cdn.jsdelivr.net/npm/@ffmpeg/core@0.12.10/dist/esm/`) the first time it's actually needed, then cached as a module-level singleton for the session. Uses the single-threaded core (not `core-mt`) so it works without COOP/COEP headers. If you bump `@ffmpeg/ffmpeg`, check that the pinned `CORE_VERSION` constant is still a compatible pairing — mismatches between the JS wrapper and the CDN-loaded core are a common ffmpeg.wasm failure mode. The `ffmpeg.exec()` timeout scales with input size (`computeTimeoutMs`: a base plus a per-MB allowance, capped at `MAX_TIMEOUT_MS`) instead of a flat constant — large-but-legitimate conversions near the size cap get more room, while a small stuck file still fails fast. Three target-specific codec quirks are hard-coded in `transcode()` and each has a reason attached — don't strip them: (1) `.webm` is pinned to VP8 (`-c:v libvpx`) because this core's default VP9 encoder reliably traps with a WASM out-of-bounds fault; (2) h264 containers (`mp4`/`mov`/`m4v`) get `-pix_fmt yuv420p` plus an even-dimension scale filter, since animated GIFs are frequently odd-sized and h264 rejects odd dimensions; (3) `apng` targets force the `apng` muxer with `-plays 0`, otherwise the `.apng` extension selects the single-frame PNG encoder and animation is lost.
- **`src/lib/converter/documentEngine.ts`** — handles the document + ebook conversions: **DOCX→PDF** (`mammoth` extracts HTML, then a manual block-layout pass writes a real text-selectable PDF via `jsPDF` — not html2canvas rasterization), **PDF→JPG/PNG** (`pdfjs-dist` renders each page to a canvas; single page → one image, multiple pages → a `jszip` archive of numbered images), **image→PDF** (`jsPDF`, one image per page, flattened onto white so transparent PNGs don't go black — HEIC/HEIF sources are transcoded to JPEG with `heic2any` first, since no browser decodes HEIC via `createImageBitmap`), **PDF→Word (DOCX)** and **PDF→EPUB** (`pdfjs-dist` extracts text per page via `getTextContent()`/`hasEOL`, reflowed into paragraphs by `extractPdfPages`, then written to an editable `.docx` with the `docx` library or packaged into a spec-valid EPUB 3 zip via `jszip`), and **EPUB→PDF** (`jszip` unzips the ebook, walks the OPF `<spine>` in reading order, inlines each XHTML's images as data URLs, and flows the combined markup through the same `renderHtmlToPdf` block-layout pass). The libraries (`jspdf`, `pdfjs-dist`, `mammoth`, `jszip`, `docx`, `heic2any`) are all dynamically imported inside this module so they stay out of the eager bundle, exactly like ffmpeg's core. `renderHtmlToPdf` recurses into structural container tags (`div`/`section`/…) so deeply-nested EPUB bodies aren't flattened into one paragraph, while mammoth's already-flat output is unaffected. Deliberate quality bounds, not TODOs: **PDF→Word and PDF↔EPUB are text-only** — they extract and reflow text, and do NOT reproduce the source's exact layout, columns, fonts, or image positions (there's no reliable fully-client-side way to do that); **PPTX→PDF stays excluded** for the same reason (no client-side renderer); and DOCX→PDF/EPUB→PDF use jsPDF's built-in Helvetica which **only covers Latin-1** (non-Latin scripts render blank until a Unicode font is embedded). `mammoth` is imported from its prebuilt browser bundle (`mammoth/mammoth.browser.js`) to avoid its Node-only deps; that subpath has no types, so a `declare module` shim lives in `src/env.d.ts`.

**`src/lib/converter/index.ts`** is the orchestrator every caller should use (`convertFile(file, targetExt, opts)`). It checks `getDocumentOp()` first and routes document conversions to `documentEngine`; otherwise it decides Canvas fast-path vs ffmpeg via `isFastPathPair()`, infers video→audio extraction (adds `-vn` to skip video decoding entirely) by comparing source/target category, rejects files over the per-category `MAX_FILE_SIZE_BYTES` cap before doing any work, and normalizes every failure into a typed `ConversionError` (`reason: 'unsupported-pair' | 'timeout' | 'engine-load-failed' | 'file-too-large' | 'unknown'`). Note document conversions can change the output extension/count (multi-page PDF→image yields a `.zip`), so the document path returns its own `filename` rather than letting `convertFile` derive it from `targetExt`.

`ConversionError` lives in its own file, **`src/lib/converter/errors.ts`**, deliberately separate from the engines. `index.ts` needs the error class at the top level but must NOT statically import anything from `ffmpegEngine.ts` **or `documentEngine.ts`**, or Vite will pull their dynamic `import(...)` calls into the eagerly-loaded chunk graph and defeat the lazy-loading (Vite warns `INEFFECTIVE_DYNAMIC_IMPORT` if this regresses; the build should stay warning-free). Both engines are only ever reached via `await import('./…')` inside `convertFile`.

**`src/lib/formats.ts`** is the single source of truth for supported formats: per-category (`image`/`audio`/`video`/`document`) `FormatDef` lists, `detectCategory(file)`, `getAvailableTargets(category, ext)`, `getDocumentOp(sourceExt, targetExt)`, `isFastPathPair()`, and `MAX_FILE_SIZE_BYTES` (per-category upload caps — image 50MB, audio 300MB, video 1GB, document 100MB — that bound worst-case conversion time rather than acting as a paywall). When adding a format, add it here — nothing else hardcodes a format list. Cross-category quirks live here too: `getAvailableTargets` merges the video→audio "extract audio" bonus targets, appends **PDF** as a target for image sources in `IMAGE_TO_PDF_SOURCES` (the Canvas-decodable set **plus `heic`**, which documentEngine decodes via heic2any), and for `document` sources returns a *curated* set (PDF→[JPG, PNG, Word/DOCX, EPUB], DOCX→[PDF], EPUB→[PDF]) rather than the media model's any-pair-within-a-category. `epub` is a `document` format (both source and target); `docx` stays `inputOnly` and is surfaced as a target only inside `getDocumentTargets` for PDF sources. Several media formats are also `inputOnly` — readable but never emitted, because writing them is either pointless or unreliable: `jfif` (byte-identical to JPEG, so it's in `CANVAS_DECODABLE_EXTS` and converts via the Canvas fast path, but we never write a `.jfif` when `.jpg` is the canonical name for the same bytes) and the MPEG-TS / camcorder / DVD video containers `mts` / `m2ts` / `ts` / `vob` / `3g2` / `f4v` (ffmpeg demuxes them reliably so `mts-to-mp4` etc. are real conversions, but muxing *back out* to them is rarely wanted and less reliable in the wasm core). `converter-controller.ts` also checks `MAX_FILE_SIZE_BYTES` at file-selection time, before rendering the format picker, so oversized files get instant feedback instead of waiting to hit the check inside `convertFile`.

Known constraint baked into `isFastPathPair`: `canvas.toBlob()` cannot actually encode BMP or GIF in any browser (it silently falls back to PNG). The fast path is intentionally narrow — decodable-source set is broad, encodable-target set is only `jpg/jpeg/png/webp`. Anything else routes through ffmpeg, which has real encoders for those containers.

**Animated images (GIF, APNG) can convert to video** and this is a load-bearing subtlety. Their MIME (`image/gif` / `image/apng`) makes `detectCategory` classify them as `image`, so a naive image target list would never offer MP4/WebM. `getAvailableTargets` therefore widens the target set for the exts in `ANIMATED_IMAGE_EXTS` to include the video containers, which is what makes `gif-to-mp4` (and the whole `gif → video` family) real conversions rather than dead pages. The invariant to preserve: `getAvailableTargets('image', 'gif')` must return the *same* targets at build time (when the SEO pages below pre-compute a preselected target) and at run time (when `converter-controller.ts` re-renders the picker after detecting the uploaded file's category). If those ever diverge, a page advertises a conversion its own picker won't offer — that exact bug is why this branch exists.

`formats.ts` also owns the data behind the per-conversion SEO pages (see Page composition): `getAllConversionPairs()` expands every `(source ext → target ext)` into a unique-slug list (`gif` lives in both the image and video lists — the pairs are deduped by slug, image wins, and the run-time category re-detection means one `gif-to-webp` page serves both), plus `getRelatedPairs()`, `getPopularPairs()` (a hand-curated slug list, filtered to slugs that actually generate), and `displayLabel()` (strips parentheticals so "GIF (animated)" → "GIF" for headings).

### UI

No UI framework is installed; every interactive piece is hand-written vanilla TS (see also `theme-toggle.ts` under Theming and the `nav-dropdown.ts` "Conversions" menu under Page composition). The converter widget is a custom element:

- **`src/components/Converter.astro`** — static markup/CSS. Visibility of each stage (`dropzone`, pick/converting/done/error) is driven entirely by a `data-state` attribute on the `<file-converter>` root, toggled via CSS descendant selectors (`file-converter[data-state="..."] .stage-x { display: block }`) — no framework, no layout-shift. Accepts optional `presetTarget` / `presetSourceLabel` / `presetTargetLabel` props (used by the per-conversion SEO pages): `presetTarget` is emitted as a `data-preset-target` attribute and the dropzone copy is tailored server-side ("Drop your PNG file…").
- **`src/scripts/converter-controller.ts`** — `customElements.define('file-converter', ...)`, the whole state machine (`idle → dragging → file-selected → converting → done/error`), wired to drag/drop, the hidden file input, and dynamically-rendered format-picker buttons (built at runtime from `formats.ts`, not duplicated as static HTML per category). When `data-preset-target` is present, the matching format chip is auto-selected after a file is chosen (expanding the collapsed "Show all formats" group if the target lives inside it), so on a `/png-to-jpg`-style page the user only has to pick a file and hit Convert.

Follow this pattern for any other non-trivial interactive component in this project rather than reaching for React/Vue — there's no framework integration configured, and adding one for a single self-contained widget isn't worth the bundle cost next to the already-lazy-loaded wasm core. The NavBar "Conversions" dropdown (`nav-dropdown.ts`) is a second, simpler instance of the same approach: a plain script that initializes any `[data-dropdown]` root (click to toggle, outside-click / Escape to close, `aria-expanded` synced).

### Styling

**No Tailwind**, despite the `tailwind-4-docs` skill being available in `.agents/skills/` — it isn't used in this project. Styling is hand-written CSS using custom properties translated 1:1 from `DESIGN.md`'s token tables:
- `src/styles/tokens.css` — every color/spacing/radius/typography/elevation token as a `--variable`.
- `src/styles/global.css` — reset + base typography, imports `tokens.css`.

[DESIGN.md](DESIGN.md) (Vercel-inspired design system) is the source of truth for all visual decisions — colors, type scale, spacing, radius, elevation, component chrome. **Read it before writing or modifying any UI.**

### Fonts

Configured via Astro's native Fonts API in `astro.config.mjs` (`fonts: [...]` using `fontProviders.fontsource()` for Geist Sans/Geist Mono), rendered with `<Font cssVariable="..." />` from `astro:assets` in `src/layouts/Layout.astro`. Astro downloads and self-hosts these at build time — there's no runtime request to a third party, which reinforces the "nothing leaves your device" story. Don't add a manual `<link>` tag or a Fontsource npm dependency; extend the `fonts` array in config instead.

### Theming

Light/dark is driven by a `data-theme="dark"` attribute on `<html>`, with `tokens.css` redefining every `--color-*` / `--shadow-*` variable under a `[data-theme='dark']` block — components never branch on theme, they just consume tokens. Three pieces coordinate:
- An **inline `is:inline` script in `src/layouts/Layout.astro`** runs before first paint, reading `localStorage.theme` (falling back to `prefers-color-scheme`) and setting the attribute. This must stay inline and in `<head>` — moving it to a module or the body reintroduces a flash-of-wrong-theme.
- **`src/scripts/theme-toggle.ts`** wires the NavBar toggle button: flips the attribute, persists to `localStorage.theme`, and syncs `aria-pressed`. It's the only place the theme is written after load.
- The toggle lives in `NavBar.astro`; `localStorage.theme` (values `'light'`/`'dark'`) is the single persisted key.

### Page composition

This is a multi-page app (Astro's default — every route is a full document, no client-side router). `src/pages/` holds one `.astro` file per route:
- `index.astro` — deliberately homepage-as-tool: `NavBar → Converter (the hero) → Popular conversions grid → FeatureStrip → HowItWorks → Footer`. The "Popular conversions" grid links into the per-conversion pages below (`getPopularPairs`). There is no separate marketing *splash* page ahead of the tool — don't reintroduce that split. (The per-conversion pages below are not that split: each one *is* a working converter, just with a specific heading and a preselected target.)
- **`[conversion].astro`** — a dynamic catch-all route whose `getStaticPaths()` statically generates **one page per conversion pair** from `getAllConversionPairs()` (~690 pages: `/png-to-jpg`, `/mp4-to-mp3`, `/gif-to-mp4`, …). This is the SEO surface — each page carries its own `<h1>`, meta title/description, and canonical, and mounts a `Converter` preset to that pair's target. It just wires props into `ConversionLanding.astro`; add a format to `formats.ts` and its pages appear automatically.
- **`conversions.astro`** — the `/conversions` index: every generated pair grouped by source category (Image / Video / Audio / Document, via `getAllConversionPairs()`) with a jump nav. Standalone file, so it's listed in `sitemap.xml.ts`'s `STATIC_PATHS`. Linked from the NavBar dropdown's "View all conversions" link.
- **`video-to-gif.astro` / `image-to-gif.astro`** — hand-written category **hub** pages. They aren't `(source ext → target ext)` pairs (there's no `video`/`image` ext), so they can't come from the matrix and are standalone files that also render `ConversionLanding.astro` with `presetTarget="gif"` and a curated related-links list. Add hubs as explicit files like these, and remember to also list them in `sitemap.xml.ts`'s `STATIC_PATHS` (the pair pages are added there automatically, standalone files are not).
- **`pdf-converter.astro` / `document-converter.astro` / `ebook-converter.astro`** — broader category hubs for the document/ebook conversions. Unlike the gif hubs these render `ConversionLanding.astro` with **no `presetTarget`** (its preset props are optional), so the visitor gets the general converter and picks any supported target after upload — they exist for the "PDF converter" / "document converter" / "ebook converter" search intent and cross-link the specific pairs. Also registered in `sitemap.xml.ts`. Note the PDF→Word pair page is at the ext-based slug **`/pdf-to-docx`** (there's no `word` ext); its `<h1>`/title still read "PDF to Word" because `displayLabel('Word (DOCX)')` → "Word", so link to `/pdf-to-docx` with a "PDF to Word" label rather than inventing a `/pdf-to-word` route.
- `about.astro`, `contact.astro`, `terms.astro`, `privacy.astro` — content/legal pages.
- `404.astro`, `500.astro` — error pages (Astro emits `404.html`/`500.html` on build for the host to serve).
- **`sitemap.xml.ts`** — an endpoint that lists `STATIC_PATHS` + every pair slug. It (and the `<link rel="canonical">` / OG tags in `Layout.astro`) depend on `site` being set in `astro.config.mjs` (`https://freeformatconvert.com`) to build absolute URLs.

**`src/components/ConversionLanding.astro`** is the shared template behind both the generated pair pages and the hubs (heading + subtitle + `Converter` + related-links grid + `FeatureStrip`/`HowItWorks`). Its `presetTarget`/`presetSourceLabel`/`presetTargetLabel` props are optional: pair pages pass them to preselect a target, category hubs omit them to show the general converter. Extend it rather than duplicating its markup into individual pages.

`Faq.astro` still exists as a component but is intentionally **not** mounted on `index.astro` right now (temporarily pulled — planned to return later). Don't delete it, and don't re-add it to the homepage without being asked.

Every page wraps its content in `Layout.astro` and shares `NavBar` + `Footer`. Both the `NavBar` and `Footer` link to the four content/legal pages (`/about`, `/contact`, `/terms`, `/privacy`) — keep those two link sets in sync. The NavBar's first item is a **"Conversions" mega-menu dropdown** (markup in `NavBar.astro`, behavior in `nav-dropdown.ts`): its columns are built server-side from `getPopularPairs()` grouped by source category, followed by the category-hub links and a "View all conversions" link to `/conversions` — so it stays in sync with `formats.ts` automatically. There's still no JS hamburger for the plain links: below 600px they wrap onto their own centered second row (see the `max-width: 599px` block in `NavBar.astro`), where the dropdown panel also re-centers itself under that row instead of anchoring to the trigger. Content pages use `PageHeader.astro` for the title/subtitle band and the `.prose` / `.button-primary` utility classes defined in `global.css` — reach for those before writing new per-page CSS, so the pages stay visually consistent.
